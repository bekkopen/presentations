Staging:

$ git add .          # add alle filer
$ git add css        # add alt i mappa css
$ git add css/*.css  # add alle css-filer i mappa css
$ git add -u .       # add endringer kun for tracka filer

Commit:

$ git commit -m  "Sjekker inn staga endringer"
$ git commit -am "Sjekker inn endringer i tracka filer"
$ git commit         # sjekk inn staga filer, editor for message
$ git commit --amend # endre på siste commit

Hvordan lagrer Git the shit?

Git thinks of its data more like a set of snapshots of a mini filesystem. Every time you commit, or save the state of your project in Git, it basically takes a picture of what all your files look like at that moment and stores a reference to that snapshot.

En git-commit:

  • Er et snapshot av filsystemet ditt
  • Ikke kun endringer fra forrige commit
  • Vet tilstanden til _alle_ filene
    • Lagrer kopi av alle endra filer
    • Linker til forrige versjon for uendra filer
  • Peker til forrige commit
  • ... og har chekcsum på ALT.

Finn et bilde

git branch

Lettvekts. Kun en peker til en commit-sha.

$ git branch greier          # opprett ny branch 'greier'
$ git checkout greier        # sjekk ut branchen

$ git checkout -b nye-greier # opprett og sjekk ut ny branch 'nye-greier'

$ git checkout master        # bytting av branch
$ git checkout nye-greier

$ find .git/refs/heads -type f
$ cat .git/HEAD

HEAD: peker alltid til siste commit i branchen du står i

git merge -ff

Fast forward: flytter pekeren

Dette er den enkleste forma for merging


$ git co -b iss53
$ git commit -am "Issue-53 fiksa"
$ git co master
$ git merge iss53 # flytt master-pekeren fram

git merge --no-ff

No fast forward: lager en merge commit

Viser eksplisitt at kode/commits har oppstått i en anna branch 


$ git co -b iss53
$ # flere commits
$ git co master
$ git pull --ff-only
$ git merge iss53 # oppretter en merge-commit
